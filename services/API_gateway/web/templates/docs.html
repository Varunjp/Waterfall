01Getting Started

CREATE AN ACCOUNT

Every job, worker, and metric in Waterfall is scoped to an App. You'll create one account, get back a permanent app_id, and use that ID on every request from then on.

1

Register

Sign up at /register with your email and a workspace name. No credit card required on the free tier.

2

Copy your App ID

Your dashboard shows a generated app_id (a UUID) the moment your account is created. This identifies your app on every API call and every worker connection.

3

Store it safely

Save it in your environment config or secrets manager. It cannot be recovered automatically from the dashboard later — only regenerated, which invalidates the old one.

!

Save your App ID immediately It will not be shown again in full after you leave the account creation screen. If you lose it, you'll need to connect Waterfall Admin, for App ID recovery.

02Identification

APP ID & AUTH

The Jobs API identifies your app via the app_id field in the request body. The Admin Dashboard and account-level endpoints instead use a bearer token, issued when you log in, and sent as an Authorization: Bearer <token> header.

Credential Used for Where it goes
app_id Enqueuing jobs, worker registration Request body field
admin_token Dashboard, account & plan management Authorization header
03Jobs API

ENQUEUE A JOB

Send a POST request to https://waterfall.varunjp.in/api/v1/jobs with your app_id, a job type, and a payload string. The payload is opaque to Waterfall — it's handed back to your worker exactly as you sent it, so encode it however your worker expects (JSON-as-string is the common case).

HTTP · JSON 
POST https://waterfall.varunjp.in/api/v1/jobs
Content-Type: application/json

{
  "app_id": "2377843c-8e5f-413c-b276-faa78c90c16f",
  "type": "email",
  "payload": "{\"to\":\"test@mail.com\",\"body\":\"Hello\"}"
}

type is your own label — it's how Waterfall routes the job to the workers registered to handle it (see Building a Worker). A job with no matching worker type stays queued until one connects.

04Jobs API

SCHEDULE FOR LATER

Add a schedule_at field with an ISO-8601 UTC timestamp to defer execution. The job sits in SCHEDULED status until that moment, then moves to the ready queue for pickup — with the same millisecond precision either way.

HTTP · JSON 
POST https://waterfall.varunjp.in/api/v1/jobs
Content-Type: application/json

{
  "app_id": "416668fb-2344-4f3d-9f26-7f1e64e248b8",
  "type": "email",
  "payload": "{\"to\":\"test@mail.com\",\"body\":\"Hello\"}",
  "schedule_at": "2026-03-21T10:12:05.146Z"
}
i

Timestamps are always UTC Convert local times before sending. The dashboard displays scheduled and created times in your browser's local timezone, but the wire format is always UTC.

05Jobs API

RESPONSE FORMAT

Both calls above return the created job record, including the retry budget assigned by your plan.

JSON 
"jobId": "3f8a91d2-9b7e-4a31-8e0f-2a6c7d4b9e10",
"status": "SCHEDULED",
"retry": 0,
"maxRetry": 3
Status Meaning
QUEUED Ready immediately, waiting for a worker
SCHEDULED Waiting for its schedule_at time
RUNNING A worker has claimed it and is executing
COMPLETED Finished successfully
FAILED Exhausted all retries
CANCELLED Cancelled before execution
06Workers

BUILDING A WORKER

Workers don't poll a REST endpoint — they open a single, long-lived bidirectional gRPC stream to the scheduler. Everything a worker needs to do is defined as a message in scheduler.proto: registering itself, sending heartbeats, reporting progress, and returning a final result.

This is also what powers the live worker view in the admin dashboard — a worker only appears as ● online there because it registered and is heartbeating over this same stream.

1

Open the stream

Call Scheduler.JobStream — a bidirectional stream of WorkerMessage sent by you and SchedulerMessage sent back.

2

Register first

Send a WorkerRegister with your app_id, the job_types you can handle, and your max_concurrency. The scheduler replies with a StreamAck.

3

Heartbeat continuously

Send a WorkerHeartbeat on an interval (every few seconds) with your current active_jobs count. Miss too many and the dashboard marks you STALE, then OFFLINE.

4

Receive assignments

The scheduler pushes JobAssignment messages as jobs of your registered types become ready, up to your declared concurrency.

5

Report the result

Optionally stream JobProgress while running, then send a JobExecutionResult with JOB_RESULT_SUCCESS or JOB_RESULT_FAILED when done.

07Workers

THE JOBSTREAM RPC

This is the full picture from scheduler.proto — a single streaming RPC carries every message type in both directions, multiplexed with a oneof.

PROTOBUF 
service Scheduler {
  rpc JobStream(stream WorkerMessage) returns (stream SchedulerMessage);
}

message WorkerMessage {
  oneof payload {
    WorkerRegister register = 1;
    WorkerHeartbeat heartbeat = 2;
    JobProgress progress = 3;
    JobExecutionResult result = 4;
  }
}

message SchedulerMessage {
  oneof payload {
    JobAssignment job = 1;
    StreamAck ack = 2;
  }
}

08Workers

RUNTIME & MONITORING

Once your workers are registering and heartbeating, the dashboard's Monitor panel populates automatically — no extra integration needed. You can also pull the same data yourself over REST.

HTTP 
GET /api/v1/runtime/overview

// returns queue depth, running jobs, and worker
// status (online / busy / stale / offline) per queue

Backed by GetTenantRuntimeResponse in the proto file — the same per-queue and per-worker breakdown shown in the dashboard's live monitor.

09SDKs

GO SDK

⚒

In active development beta soon A first-class Go SDK that wraps the worker handshake, heartbeat loop, and job dispatch is currently being built and isn't published yet. In the meantime, generate a client directly from scheduler.proto with protoc — the wire contract is stable and won't change underneath you.

SHELL 
# generate a Go gRPC client from the proto file
protoc --go_out=. --go-grpc_out=. scheduler.proto

# then implement WorkerRegister → heartbeat loop →
# handle JobAssignment → send JobExecutionResult

Once published, the SDK will expose this same flow as a small waterfall.Worker interface — register a handler per job type and the client manages the stream, heartbeats, and retries for you.
